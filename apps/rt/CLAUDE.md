# apps/rt ŌĆö `mustard-rt`

Subproject guard for the Mustard runtime crate. Auto-loaded when working under `apps/rt/`.

## What this is

The `mustard-rt` crate ŌĆö the single enforcement + automation binary. It has three
faces:

- **Hooks** (`src/hooks/`) ŌĆö one module per concern (`bash_guard`, `budget`,
  `close_gate`, `model_routing`, `tracker`, ŌĆ”). `settings.json` wires one
  `mustard-rt on <event>` entry per lifecycle event; `dispatch.rs` runs every
  module registered for that event.
- **Run subcommands** (`src/run/`) ŌĆö `mustard-rt run <name>`: `sync-detect`,
  `sync-registry`, `scan-orchestrate`, `scan-finalize`, `verify-pipeline`,
  `security-scan`, and more. These replaced the old JS `scripts/` payload.
- **MCP** (`src/mcp/`) ŌĆö the Rust MCP server face.

## Build & test

```bash
cargo build -p mustard-rt
cargo test  -p mustard-rt
cargo run   -p mustard-rt -- run sync-detect      # discover subprojects (JSON)
cargo run   -p mustard-rt -- run scan-orchestrate # /scan pre-dispatch
```

## Guards

- **Fail open.** Hook dispatch must never abort user work ŌĆö per-step errors are
  reported inside the JSON result; the process exits `0`.
- **Run subcommands emit byte-stable JSON.** The pipeline parses stdout; keep
  field order and the two-space pretty layout when porting/altering output.
- Assets a binary must read but does not distribute (e.g. the scan agent-prompt
  template) are embedded via `include_str!`, with an on-disk copy as optional
  override ŌĆö do not reintroduce hard file dependencies.
- Code, comments and doc-comments in EN; surgical edits only.
- `unsafe` is forbidden (`#![forbid(unsafe_code)]`); `unwrap_used` is `deny` in non-test code.
- Add enforcement modules by appending to `Registry::new()` ŌĆö never modify the dispatcher.
- `rusqlite` pinned to `0.31` ŌĆö matches dashboard `src-tauri` to share `libsqlite3-sys`.
- Subprocess calls: `cmd /C` on Windows, `sh -c` on POSIX.
- Telemetry: always `let _ = store.append(...)` ŌĆö never load-bearing.

## Statusline themes

`mustard-rt run statusline` reads `MUSTARD_STATUSLINE_THEME` (defaults to `catppuccin`). Accepted values: `default`, `minimal`, `tokyo-night`, `catppuccin`, `pastel-powerline`, `gruvbox-rainbow`. Powerline themes require a Nerd Font; users without one should set `MUSTARD_STATUSLINE_THEME=default`. Preview all themes with `mustard-rt run statusline --preview`.

## Scan subsystem (project-profiler Wave 2)

There is **no per-language scanner** under `src/run/scan/` any more. Wave 2 of
`project-profiler` removed the eight hand-written `*_scanner.rs` files
(`dotnet`, `typescript`, `dart`, `php`, `python`, `java`, `go`, `rust`) and
replaced them with a single generic interpreter:

1. `scan::file_utils::visit` performs the single-pass parallel read (Wave 1).
2. `scan::cluster_discovery::discover_clusters` mines structural clusters
   from the visited file set ŌĆö fully agnostic, no per-language code.
3. `scan::interpret::interpret` (cold path, cached per-subproject via
   `.claude/.interpret-cache.json`) asks a model to label clusters, identify
   entities, and emit `[[ ]]` edges. **Fail-open**: no API key / network
   error / parse failure ŌćÆ empty interpretation, the agnostic cluster output
   is the floor.

Env knobs:

- `MUSTARD_SCAN_MODEL` ŌĆö `sonnet` (default), `opus`, `haiku` (opt-in only ŌĆö
  no silent downgrade; unknown values resolve up to Sonnet).
- `ANTHROPIC_API_KEY` ŌĆö required for the model round-trip; absent ŌćÆ no-op.
- `MUSTARD_INTERPRET_CACHE=off` ŌĆö bypass both read + write of the interpret
  cache (used by `interpret_cache_frozen` to assert the freeze).

Adding a new language is no longer "write a new `*_scanner.rs`". The
interpreter is language-agnostic; the only place a language id leaves a
trace is `STACK_SIGNALS` in `scan/mod.rs` (manifest sniff for
`detect_stack`) and `primary_ext_for_stack` in `project_conventions.rs`
(used by the agnostic cluster discovery to pick which extension to walk).
