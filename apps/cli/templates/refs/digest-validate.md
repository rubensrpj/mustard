# Digest validation â€” the shared AI step after the scan digest

Loaded on demand by **every** flow that researches through the scan digest: `/feature`, `/task`, `/bugfix`. **Single source of truth â€” never copy this prose into a SKILL; reference this file.** (SOLID/DRY: one definition, many references.) For HOW to locate code in the first place â€” semantic search (mustard) vs `grep`/`glob` â€” see [[locating-code]].

## Why
The scan is deterministic: it LOCATES, it cannot JUDGE. Three judgements it cannot make, but a model can from the digest answer alone:
1. **Is an anchor a REAL target or an incidental lexical match?** A UI request matching a backend credit-card file on the bare word `card` is noise â€” and no deterministic score removes it (field-proven: a `stratum_floor` A/B left the anchors identical, because the noise is *meaning*, not rank).
2. **Does the work actually need the feature pipeline, or is it a lean `/task`? And if a feature â€” light or full?** The deterministic `layerCount` OVER-counts: it spans every project an anchor touches, including the incidental ones.
3. **Did the CENTRAL concept actually get FOUND, or did only common vocabulary match?** Field-proven failure: a query for "efetivar previsÃ£o" came back `reason=strong` because common terms (payable/valor/data) matched, but the central concept `efetivarâ†’effectivate` hit tier `none` â€” so the digest ranked the WRONG file and the validator (blind to the miss) blindly confirmed it. The render now surfaces a `## MISSED / WEAK CONCEPTS` section so the model can judge whether a missed term is the central concept; `reason` alone is NOT trustworthy when it is.

So one model step, ONE layer above the scan, validates the answer before any flow acts on it.

## When to run it (gate)
Run it after the digest returns a usable answer (`report.reason` âˆˆ {`strong`, `weak`}). It is **NOT optional on a `strong` digest** â€” `strong` measures match COUNT, not whether the target landed in the RIGHT layer. The highest-risk case is exactly a `strong` answer whose top anchors are layer-INCOHERENT with the request (a frontend UI task ranking backend validators/DTOs at the top because the shared domain nouns matched there) â€” that is the #1 noise mode (Why #1) this step exists to `dropped`. Do NOT spend an Explore on, or implement from, layer-incoherent anchors before validating. **SKIP** only when the render is empty (no concept matched â€” nothing to validate) or the digest `miss`ed entirely after the repo-vocabulary re-query (true net-new â†’ treat as DESIGN, not recomposition).

## Steps
1. **Render the prompt (deterministic).** `mustard-rt run digest-validate-render --intent "<INTENT>" --model .claude/grain.model.json`. Pure deterministic â€” no model call inside the rt: it REUSES the feature digest's retrieval AND tags each anchor with its project (`read_projects`), so the validator sees EXACTLY the concepts/anchors the digest surfaced, each with its layer. Stdout = the raw prompt; **empty stdout = skip the validation and fall through to the flat pruned anchors** (never dispatch on an empty prompt).
2. **Dispatch the validator (Sonnet).** Pass the rendered prompt VERBATIM as the Task prompt to `subagent_type: general-purpose` with `model: sonnet`. **Sonnet, not Haiku:** this is ONE routing-critical call per pipeline entry (not a fan-out), so accuracy outweighs the negligible per-call cost â€” a wrong `full` is the most expensive routing error. The validator replies with ONLY a JSON object:
   `{"route":"task|feature","scope":"light|full|","dropped":["<file>"],"concerns":[{"label":"<short>","concepts":["<concept>"],"anchors":["<file>"]}],"centralFound":true|false,"requeryBridges":[{"userWord":"<missed user-side word>","codeTerms":["<real code identifier>"]}]}`
   - `centralFound: false` means the most-discriminative concept of the intent landed under MISSED / WEAK â€” the kept anchors matched only common vocabulary and point at the WRONG flow. `requeryBridges` is then the PAIRED bridge(s) from each missed USER-SIDE word to the real ENGLISH code identifiers it maps to (e.g. `{"userWord":"efetivar","codeTerms":["effectivate","settle","confirm"]}`) â€” a pair, not a flat list, so the persist knows `efetivar â†’ effectivate`, never a cartesian. Both default to `true` / empty when absent (an old reply carries no retrieval concern).
3. **Act on the verdict.**
   - **`centralFound: false` â†’ EMBEDDING-SEARCH first (local vector recall), then RE-QUERY + PERSIST (auto-heal).**
     0. **Embedding recall â€” the recall the name-index structurally cannot reach.** Run `mustard-embed search --intent "<the missed central concept, TRANSLATED TO ENGLISH>" --vectors .claude/grain.vectors`. It returns the files whose method BODIES mean the concept, ranked by similarity â€” exactly the methods whose NAME diverged from the request (field-measured: PT `efetivar`â†’`EffectivateAsync`, `dar baixa`â†’`WriteOffAsync` â€” name-match recall 0/10). Recall, **not precision**: on a large mixed repo the right file competes with interfaces/status services/display components, so it can sit at #2â€“#5. Each hit carries `method`+`line`: **read those candidate spans (top ~3-5) and pick the one that PERFORMS the action** â€” this one-read re-rank IS the precision step (recovers the per-method judge's role at ~zero scan cost). Requires a vectors index (`mustard-embed build`, built by `/scan`); an empty result or an absent `mustard-embed` â†’ fall through to the bridge re-query below. See [[scan-enrich-purpose]].
     1. **Re-query** the digest with the corrected vocabulary â€” flatten the `codeTerms` across `requeryBridges` and run `mustard-rt run feature --intent "<all codeTerms joined>"`. Use ITS deterministic anchors for the right flow; only THEN proceed to `dropped`/`route`/`scope` against the re-queried answer. A `strong` reason is **NOT** trustworthy when the central concept missed: `centralFound` OVERRIDES the aggregate `reason`. (Re-query, do not just drop â€” the right files were never in the first answer.)
     2. **Persist ONLY when DOUBLY confirmed.** A `strong` re-query proves the term EXISTS, NOT that it means the RIGHT thing â€” `import` matches the real `ImportOfxAsync` flow (an OFX bank-statement import â€” a DIFFERENT feature, NOT a programming keyword) and `user` matches the auth/login user, both `strong` in the WRONG sense, and would poison the lexicon (field-proven: `importacaoâ†’import`, `colaboradorâ†’user`). So persist only when BOTH hold: (a) the re-query `report.reason` is `strong`, AND (b) its top anchors land in the SAME domain the request is about (they overlap the concept's kept anchors / its concern), not a different flow. Then write `requeryBridges` (already in the `{userWord, codeTerms}` proposal shape) to a temp file and run `mustard-rt run lexicon-enrich --apply <file>`, surfacing a one-line note (e.g. `auto-bridged: efetivarâ†’effectivate (confirmed)`). The next identical query then hits the deterministic `lexicon` tier â€” no LLM call.
     3. **Do NOT persist** when the re-query is not `strong`, OR when it lands on a DIFFERENT flow than the request (the term matched in the wrong sense). When unsure, PROPOSE rather than auto-write â€” surface the bridge + its re-query anchors and let the orchestrator/user confirm (the `lexicon-suggest --accept` pattern). A wrong bridge mis-directs EVERY future query, so silence beats a bad write.
   - `centralFound: true` (or absent) â†’ proceed normally.

   **Principle:** the semantic judge is the SOURCE of new bridges; the lexicon is a CACHE the judge populates on confirmed hits â€” so the deterministic path covers more vocabulary over time and the LLM fires only on genuinely-new words. Two guards keep the cache clean: the judge returns an EMPTY `requeryBridges` for net-new concepts and same-word-different-sense matches (a feature to BUILD is not a word to bridge), and the persist gate writes only when the re-query lands in the request's own domain â€” a `strong` re-query proves a term EXISTS, never that the mapping is RIGHT. The `--apply` anti-hallucination gate only checks the target is mined; it does NOT check sense, which is why the domain check above is required.
   - **`dropped`** â†’ remove those anchors from what you read; they are incidental (a far-layer / ambiguous-term match). Never open them.
   - **`route` = `"task"`** â†’ the real work is single-layer and small: it does NOT need the pipeline. Run it as **`/mustard:task`** (lean â€” no spec, no wave, no QA gate) on the KEPT anchors, and STOP the heavier flow. *(This is the âˆ’81%-turns lever: most enhancements land here.)*
   - **`route` = `"feature"`** â†’ continue the feature pipeline. **`scope` = `"light"`** â†’ inline EXECUTE; **`scope` = `"full"`** â†’ open `feature/full-plan.md` and follow PLAN.
   - **`concerns`** â†’ each object is its OWN unit of work, scoped to its `anchors` (never the flat list): `/feature` â†’ a unit in the three-natures split (Full: a wave); `/task` â†’ its own dispatched action; `/bugfix` â†’ its own diagnose+fix.
4. **Deterministic fallback (never block the flow).** If the validator is unavailable, the dispatch errors, or its reply does not parse into an object with a `route`, DROP the verdict and proceed from the flat pruned `anchors` + the DETERMINISTIC scope (fill the census â†’ `spec-draft` â†’ `plan-prepare`) exactly as you would without it. The validator only ever SHARPENS â€” its absence degrades to the deterministic path, never to a stall.

## INTENT hygiene â€” the rule that makes or breaks the validator
**`<INTENT>` MUST be the user's actual REQUEST â€” their own words / phrasing of WHAT they want â€” the SAME rich intent you passed to the digest (the user's content words + your code-vocabulary translation), NOT a stripped list of isolated code terms.** The validator judges meaning (is this `card` the UI card or the credit card?); given only bare terms it loses the context that disambiguates, and prunes/routes wrong. Never lapidate the request away before the validator.

## Invariant
The scan digest is **100% deterministic**; the validator is the **ONLY AI step** and lives in the **ORCHESTRATION**, never inside the scan. `digest-validate-render` does pure deterministic assembly (reuses the digest retrieval + project span, emits a byte-stable prompt, calls no model); the JUDGEMENT is the dispatched Sonnet agent's, run one layer above the scan. It SUBSUMES the older concern-split judge â€” its `concerns` field carries the same partition, so flows run THIS step, not a second judge.
