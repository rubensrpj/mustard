# Locating code — literal → grep, concept → digest, rules → Guards

Loaded by every flow that finds code (`/feature`, `/task`, `/bugfix`, and ad-hoc location). **Single source of truth — reference this file, never copy the prose.** Pick the search tool by WHAT you already know:

- **You are ORIENTING (which subproject? where are the entry points?)** → the terrain is ALREADY in your window. The orientation census (`mustard-rt run orient`) is injected at session start (the subprojects, each `name · kind · files · layer`) and per request (the files your request's concept matches — ranked by corpus rarity, grouped by subproject, as `path:line`). Read those entry points first — never grep to orient yourself.
- **You know a LITERAL token (an exact symbol, an error string, a file glob)** → use `grep`/`glob`. For a token you already have they are exact, instant and complete — do not spend a digest call on it.
- **You know the CONCEPT but not the name (vocabulary may diverge — "settle a payable" → `WriteOffAsync`, "efetivar" → `EffectivateAsync`)** → use the deterministic digest: `mustard-rt run feature --intent "<the request + your code-shaped translation>"` — a BM25 ranking over the mined `grain.model.json`, no model call. It returns the anchors (~12 real files) with per-anchor provenance; **the digest finds WHERE to look, it does not replace reading** — always READ the files it points to. On a `weak`/`none` report it returns the `candidates` term index (the repo's REAL vocabulary): sharpen your terms against it and re-call — a `miss` is NOT "absent".
- **You need the LAYER's rules (conventions, do/don'ts) rather than a location** → the subproject's `## Guards` in its `CLAUDE.md` — auto-loaded the moment you work on files under that subproject; no pipeline needed.

**Shaping the digest query — lead with the WORK, not the DATA.** When the task is bound to ONE layer (a frontend datatable cell, a migration, an endpoint), lead the `--intent` with the terms that name the WORK/layer (`datatable`, `column`, `cell`, `wrap`, `tooltip`, `badge`, `render`) over the shared DOMAIN nouns the data carries (`cpf`, `cnpj`, `document`, `validation`). A domain noun matches in EVERY layer, and the plumbing layer (validators, DTOs, services) usually has MORE occurrences of it — so a data-noun-led query lets the wrong layer out-score your target on raw frequency while the real target sinks. The work/structure vocabulary DISCRIMINATES the layer; the domain noun is the noise that drags in the wrong-layer plumbing.

Digest recall is strong but NOT perfect — ALWAYS verify by READING the returned candidates; never act on the top hit blind. The `grain.model.json` index is built by `/scan`; if it is absent, run `/scan` first — grep alone cannot bridge a vocabulary gap.
