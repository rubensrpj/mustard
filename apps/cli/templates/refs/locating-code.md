# Locating code — semantic-first, grep only for literals

Loaded by every flow that finds code (`/feature`, `/task`, `/bugfix`, and ad-hoc location). **Single source of truth — reference this file, never copy the prose.** Pick the search tool by WHAT you already know:

- **You are ORIENTING (which subproject? where are the entry points?)** → the terrain is ALREADY in your window. The orientation census (`mustard-rt run orient`) is injected at session start (the subprojects, each `name · kind · files · layer`) and per request (the files your request's concept matches — ranked by corpus rarity, grouped by subproject, as `path:line`). Read those entry points first — never grep to orient yourself. The census matches concepts that live in file PATHS; reach for the semantic search below for a concept BEYOND that (a symbol-only or PT→EN vocabulary gap it cannot bridge).
- **You know the CONCEPT but not the name (vocabulary may diverge — "settle a payable" → `WriteOffAsync`, "efetivar" → `EffectivateAsync`)** → use mustard's SEMANTIC search, never a blind grep (grep cannot bridge a vocabulary gap). In a pipeline the digest (`mustard-rt run feature --intent "<the request>"`) is the research step; for a direct ad-hoc lookup, `mustard-embed search --intent "<concept, IN ENGLISH>" --vectors .claude/grain.vectors` is a first-class locate tool — the index is English-canonical, so translate the concept to English first. It returns the files whose method BODIES mean the concept, each with `method`+`line`; read those spans to confirm.
- **You know a LITERAL token (an exact symbol, an error string, a file glob)** → use `grep`/`glob`. For a token you already have they are exact, instant and complete — semantic search is slower and fuzzier there, so do NOT use it for literals.

**Shaping the semantic query — lead with the WORK, not the DATA.** When the task is bound to ONE layer (a frontend datatable cell, a migration, an endpoint), lead the `--intent` with the terms that name the WORK/layer (`datatable`, `column`, `cell`, `wrap`, `tooltip`, `badge`, `render`) over the shared DOMAIN nouns the data carries (`cpf`, `cnpj`, `document`, `validation`). A domain noun matches in EVERY layer, and the plumbing layer (validators, DTOs, services) usually has MORE occurrences of it — so a data-noun-led query lets the wrong layer out-score your target on raw frequency and inflates `reason=strong` on COUNT while the real target sinks. The work/structure vocabulary DISCRIMINATES the layer; the domain noun is the noise that drags in the wrong-layer plumbing.

Semantic recall is strong but NOT perfect (~70-90% on hard cross-vocabulary cases) — ALWAYS verify by READING the returned candidates; never act on the top hit blind. The vectors index is built by `/scan`; if it is absent the semantic search is empty — fall back to the digest, then grep.
