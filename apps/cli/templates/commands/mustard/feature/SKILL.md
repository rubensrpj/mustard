# /feature - Feature Pipeline

`/feature <request>` ÔÇö understand the client, research the repo through the deterministic `scan` tool (never reading source by hand), then plan + implement.
Heavy lifting delegates to `mustard-rt`; the orchestrator routes phases + emits events. You (the AI) do the reasoning that grain cannot: elicitation, decomposition, lapidation.

> **Artifact order ÔÇö read before editing the flow.** The spec dir + `spec.md` + `meta.json` are MATERIALISED by `mustard-rt run spec-draft` in PLAN (step 2). They do **not** exist during ANALYZE. ### 2. DECOMPOSE

From the insumos + anchors, split the request into three natures (this is the judgement grain cannot make):

- **Units with precedent** Ôćĺ each maps to a matched slice. Only a **net-new** unit (it CREATES an entity that does not exist yet) gets a `scan spec` compile (with `--like <sibling>` when one exists) ÔÇö the compiler emits a *create* mold. An **enhancement** unit (modifies an existing entity/behavior) skips `scan spec` and consumes the feature digest's anchors instead (the `context_enrichment` that `spec-draft` pre-fills into the scaffold's `context` section).
- **Cross-cutting invariants to obey** Ôćĺ contracts/hubs the repo already enforces (e.g. an injected `ICurrentTenant`); pass each via `scan spec --invariant <Name>` so the draft anchors the real wiring. NEVER invent the mechanism ÔÇö mirror the anchored consumers.
- **Net-new gaps** (no precedent; `miss` after a repo-vocabulary re-query) Ôćĺ surface as a design decision; do not let a `scan spec` draft's "deterministic" framing imply a unit that has no precedent is safe to clone.
