Achar o `mustard.json` sozinho **O que houve:** rodando de dentro do submódulo, ele devolveu `no-base-prefix` — não reconheceu `dev_` como prefixo válido, porque o `mustard.json` (que define quais são as bases) só existe na raiz. **O que deveria fazer:** subir a árvore de diretórios até encontrar o arquivo de configuração. É o comportamento padrão de qualquer ferramenta de projeto — o `git` sobe até achar `.git`, o `node` sobe até achar `package.json`. Exigir que você esteja no diretório exato é regressão de usabilidade. O agravante é o diagnóstico enganoso: a mensagem culpou o **nome da branch**, quando o problema era **onde o comando foi invocado**. Reconhecer submódulo como repositório **O que houve:** apontado para o submódulo, respondeu `not-a-git-repo`. **A causa:** dentro de um submódulo, `.git` é um **arquivo** com um ponteiro (`gitdir: ../../.git/modules/...`), não uma pasta. O comando verificou se existe uma *pasta* `.git` e concluiu que ali não havia repositório. **O que deveria fazer:** perguntar ao próprio git em vez de inspecionar o sistema de arquivos. `git rev-parse --git-dir` resolve gitfile, worktree e submódulo de forma transparente — é para isso que existe. Esta é a falha mais séria, e por um motivo específico: o comando `/git` declara como **regra de ferro** que "submódulos vêm antes do pai, sempre". O motor que executa o ritual é incapaz de abrir um submódulo. A promessa e a implementação se contradizem — e num monorepo com submódulo, que é exatamente este projeto, o ritual nunca completa. ## 3. Não fingir que terminou Este é o problema estrutural, e o mais perigoso. O comando na raiz retornou `"ok"`, com `"action": "settled"`, `"branchDeleted": true`, `"remoteDeleted": true` e — o detalhe crítico — **`"alsoMergeable": []`**, que significa "não há mais nada pendente". Mas havia: o submódulo continuava na branch de trabalho, com branch local e remota vivas. O relatório dizia "concluído" para um ritual que executou metade. **O que deveria fazer:** enumerar todos os repositórios que participam da unidade de trabalho e reportar o estado de cada um. Algo como: ``` { "raiz": { "settled": true }, "submodulo": { "settled": false, "reason": "unsupported-gitfile" }, "complete": false } ``` Um relatório parcial que se apresenta como completo é pior que um erro — erro você vê; falso "pronto" você só descobre quando algo quebra depois. ## Sobre o terceiro caso: ele acertou O `"reason": "dirty-tree"` que impediu atualizar a `dev` local **não é defeito**. O que faltou foi **visibilidade**: isso voltou como um campo enterrado no meio do JSON (`updated: false`), com o mesmo peso visual de qualquer outro dado. Uma pendência que exige ação humana deveria sair como pendência declarada, não como propriedade discreta de um objeto. Um refinamento útil seria distinguir dois casos: os arquivos sujos **colidem** com o que vem do remoto (aí realmente precisa da sua decisão) ou não colidem (aí o avanço rápido seria seguro mesmo com a árvore suja). --- ## A raiz comum Os três apontam para a mesma lacuna: **a ferramenta foi construída presumindo o caso simples — um repositório, árvore limpa, invocado da raiz.** O projeto aqui é o caso composto: dois repositórios, submódulo, sessões concorrentes deixando a árvore suja.
