## Audit-1: `metrics wave-status` joins on `payload.pipeline` but writers persist `spec`

- Class: b (aggregator) + a (writer never sets `payload.pipeline` for child waves)
- Source: `apps/rt/src/run/metrics_wave_status.rs:175-223` (queries) <-> live `events` rows where the `spec` column carries the *parent* spec name and `payload.pipeline` is absent
- Symptom: every wave reports `tokens_saved=0`, `duration_ms=0`, `retries=0`, `status=null`. The only producer that sets `payload.pipeline` is `apps/rt/src/run/memory.rs:131,151` (agent.memory events). `token.saved` is not emitted by *any* code path — only `rtk.savings` / `prompt.economy.saved` / `hook.savings` / `routing.savings` are (see `packages/core/src/projection/workspace.rs:73-80`). Same shape mismatch makes `cross-wave memory` injection return empty for this spec — events get recorded with `spec='2026-05-20-mustard-wave-network-standard'` but the reader filters on `payload.pipeline='wave-N-…'`.
- Fix: out of scope for this wave (apps/rt/). Recorded for follow-up: the rt query should be `WHERE spec = ?1 AND (json_extract(payload,'$.wave_name')=?2 OR json_extract(payload,'$.pipeline')=?2)` AND writers should populate one of those two keys when the active pipeline is a child wave. Dashboard wire-up below assumes the eventual fix lands — when it does, the same Tauri command starts returning real numbers without further dashboard change.
- Status: [x] documented (wave-4 cannot apply the apps/rt fix; tracked as `metrics_wave_status_writer_mismatch` follow-up).

## Audit-2: Economia page never consumes `metrics wave-status`

- Class: c (renderer)
- Source: `apps/dashboard/src/pages/Economia.tsx` (pre-fix) <-> `apps/rt/src/run/metrics_wave_status.rs` (unused)
- Symptom: AC-4 fails — the page has zero callers of `dashboardMetricsWaveStatus` or `metrics_wave_status`. The Economia page renders RTK / measured / prevention / routing / phases / promptEconomy without any parent→waves breakdown, so even when wave-status returns honest numbers nobody sees them.
- Fix: add a typed wrapper `dashboardMetricsWaveStatus(repoPath, specName)` in `lib/dashboard.ts`, add a `dashboard_metrics_wave_status` Tauri command in `spec_views.rs` + `main.rs` (lib.rs), and render the parent→waves block on `Economia.tsx` for the active spec.
- Status: [x] fixed — see `lib.rs` / `spec_views.rs` / `lib/dashboard.ts` / `pages/Economia.tsx` changes for wave 4.

## Audit-3: SpecsList renders parents and children flat (no visual grouping)

- Class: c (renderer)
- Source: `apps/dashboard/src/components/SpecsList.tsx:140-175` <-> `dashboard_specs` in `lib.rs:464-531` (which *does* emit `parent` per row)
- Symptom: AC-5 fails — `SpecsList.tsx` does not reference `parent` / `Parent` anywhere, so wave children render flat alongside standalone specs. The backend already supplies `parent` (FS walk in `specs_from_fs`), but the renderer drops it. Sorting *does* group by parent name (lib.rs:523-529), so children are adjacent, but there is no indent, badge, or visual hierarchy.
- Fix: in `SpecsList.tsx`, indent rows whose `spec.parent != null` and add a `+N waves` badge on parents that have at least one child in the current list.
- Status: [x] fixed.

## Audit-4: SpecCard has no parent affordance

- Class: c (renderer)
- Source: `apps/dashboard/src/components/specs/SpecCard.tsx`
- Symptom: A spec that is a wave-plan parent renders identically to a leaf spec. No badge advertises "this spec has waves" and there is no link to the Network tab of the drill-down.
- Fix: accept an optional `childWaves?: number` prop. When `> 0`, render a `+N waves` link badge that navigates to the spec's Network tab. Backwards-compatible default: undefined → no badge.
- Status: [x] fixed.
