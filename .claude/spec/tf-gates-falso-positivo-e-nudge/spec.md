## Contexto

Consolidação do `/btw` de campo (sialia, 2026-07-09, run real do prompt sialia-partners): os gates abaixo têm histórico de disparo falso — "um gate que precisa de escape hatch rotineiro não é gate, é atrito" — e um hook do próprio harness contradiz a regra de eficiência do CLAUDE.md. Itens 1-3 já apareceram em memórias anteriores; este TF os consolida com reprodução e critério de fechamento.

- [x] T5 **gate de aprovação forjável** (campo, sialia 2026-07-09): no fluxo Full, o próprio orquestrador rodou `mustard-rt run approve-spec` e destravou os gates SEM sinal humano — um gate que o gateado abre sozinho não é gate. Fix: `approve-spec` passa a exigir um MARCADOR de aprovação real (`<spec>/.approved-by-user`) escrito por um observer de AskUserQuestion — o hook vê a RESPOSTA efetiva do usuário no tool_response, que o modelo não fabrica; modo `MUSTARD_APPROVAL_MODE=strict|warn|off` (default strict). Em job background sem usuário: sem marcador → Full PARA no fim do PLAN com `needs input` apresentando o plano (o comportamento correto — nunca auto-aprovar). Higiene associada: a memória envenenada do sialia ("Feature pipeline: pular aprovação") deve ser removida — ela registrou uma inferência ruim como preferência e serviu de álibi, contradizendo a preferência real registrada ("ajuste não é aprovação").
- [x] T6 **Review Gate "No staged changes" falso** (campo sialia 2026-07-09): o commit gate (`bash_command_gate.rs` `review_gate`, ~linha 1454) alerta "No staged changes detected" quando dispara ANTES de um `git add` encadeado no MESMO comando (`git add -A && git commit …` → índice vazio no PreToolUse). `commit_stages_inline` (1186-1196) só cobre `-a/-am/--all`. Fix: reconhecer `git add` encadeado (`&&`/`;`) antes de acusar índice vazio; manter o aviso quando não há add algum. Teste dos dois lados.
- [x] T7 **worktree honra a base + scope-classify diagnosticável** (defeito #2 do campo): o binário JÁ está correto — `emit-pipeline` honra `--base` (`resolve_base`, testado) e grava `{base}_{slug}`; o gap é a CRIAÇÃO do worktree via `EnterWorktree` (ferramenta do harness, fora do Mustard), que corta do ref DEFAULT — o template já prescreve o workaround manual. Escopo desta TF: tornar o `scope-classify` `error-fallback` (`scope_decompose.rs:704`, spec ilegível/cwd errado) um ERRO diagnosticável (path+cwd na mensagem) em vez de devolver `full` silencioso e mentiroso. Um comando determinístico de criação-de-worktree fica FORA desta TF (unidade própria, decisão do dono).
- [ ] T8 **specs invisíveis no dashboard sob worktree** (campo 2026-07-10): após introduzir o worktree por unidade de trabalho, specs criadas trabalhando DENTRO do worktree somem do dashboard. Mecanismo (confirmado empiricamente — `tf-gates` AUSENTE no `.claude` raiz, presente no do worktree; raiz 70 specs vs worktree 65 = estado FRAGMENTADO): o worktree tem seu PRÓPRIO `.claude/` porque `git worktree add` faz checkout da árvore e `.claude/spec` é tracked; `mustard-rt` com cwd no worktree resolve o anchor pro `.claude` do worktree, e o dashboard lê o `.claude` do monorepo raiz → invisível. Diagnóstico-chave: o "fork A vs B" é FALSO — a correção é HÍBRIDA, dividida pela linha tracked/efêmero. (i) `spec.md`/`meta.json` são TRACKED por-branch → vivem LEGITIMAMENTE no worktree (o gap 70-vs-65 é divergência de branch, não bug); (ii) estado EFÊMERO (gitignored: `.events`, `.session`/markers active-spec, `.harness`, `.metrics`) NÃO deve fragmentar. Fix: (A) efêmero → CENTRAL via `main_checkout_root` (plumbing `git rev-parse --git-common-dir` JÁ existe em `git_settle.rs:58`; `set_var` proibido no edition 2024, então redirecionar em `workspace.rs:191`/`context.rs:28`, não via env); (B) spec.md fica por-branch + dashboard AGREGA (`dashboard_specs_impl` em `apps/dashboard/src-tauri/src/lib.rs:728` une raiz + `git worktree list`; hoje cada worktree vira card órfão porque `discovery.rs:17` SKIP_DIRS não pula `worktrees`). As duas metades se complementam (marker central só fica consistente com o dashboard agregando). AC: anchor sob cwd de worktree → efêmero no checkout principal; dashboard lista specs de worktree sob o projeto pai; regressão.
