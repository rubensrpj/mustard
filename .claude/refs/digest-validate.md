# Digest validation — the shared AI step after the scan digest

Loaded on demand by **every** flow that researches through the scan digest: `/feature`, `/task`, `/bugfix`. **Single source of truth — never copy this prose into a SKILL; reference this file.** (SOLID/DRY: one definition, many references.)

## Why
The scan is deterministic: it LOCATES, it cannot JUDGE. Three judgements it cannot make, but a model can from the digest answer alone:
1. **Is an anchor a REAL target or an incidental lexical match?** A UI request matching a backend credit-card file on the bare word `card` is noise — and no deterministic score removes it (field-proven: a `stratum_floor` A/B left the anchors identical, because the noise is *meaning*, not rank).
2. **Does the work actually need the feature pipeline, or is it a lean `/task`? And if a feature — light or full?** The deterministic `layerCount` OVER-counts: it spans every project an anchor touches, including the incidental ones.
3. **Did the CENTRAL concept actually get FOUND, or did only common vocabulary match?** Field-proven failure: a query for "efetivar previsão" came back `reason=strong` because common terms (payable/valor/data) matched, but the central concept `efetivar→effectivate` hit tier `none` — so the digest ranked the WRONG file and the validator (blind to the miss) blindly confirmed it. The render now surfaces a `## MISSED / WEAK CONCEPTS` section so the model can judge whether a missed term is the central concept; `reason` alone is NOT trustworthy when it is.

So one model step, ONE layer above the scan, validates the answer before any flow acts on it.

## When to run it (gate)
Run it after the digest returns a usable answer (`report.reason` ∈ {`strong`, `weak`}). **SKIP** when the render is empty (no concept matched — nothing to validate) or the digest `miss`ed entirely after the repo-vocabulary re-query (true net-new → treat as DESIGN, not recomposition).

## Steps
1. **Render the prompt (deterministic).** `mustard-rt run digest-validate-render --intent "<INTENT>" --model .claude/grain.model.json`. Pure deterministic — no model call inside the rt: it REUSES the feature digest's retrieval AND tags each anchor with its project (`read_projects`), so the validator sees EXACTLY the concepts/anchors the digest surfaced, each with its layer. Stdout = the raw prompt; **empty stdout = skip the validation and fall through to the flat pruned anchors** (never dispatch on an empty prompt).
2. **Dispatch the validator (Sonnet).** Pass the rendered prompt VERBATIM as the Task prompt to `subagent_type: general-purpose` with `model: sonnet`. **Sonnet, not Haiku:** this is ONE routing-critical call per pipeline entry (not a fan-out), so accuracy outweighs the negligible per-call cost — a wrong `full` is the most expensive routing error. The validator replies with ONLY a JSON object:
   `{"route":"task|feature","scope":"light|full|","dropped":["<file>"],"concerns":[{"label":"<short>","concepts":["<concept>"],"anchors":["<file>"]}],"centralFound":true|false,"requeryBridges":[{"userWord":"<missed user-side word>","codeTerms":["<real code identifier>"]}],"requeryTerms":["<code-side word to re-query>"]}`
   - `centralFound: false` means the most-discriminative concept of the intent landed under MISSED / WEAK — the kept anchors matched only common vocabulary and point at the WRONG flow. `requeryBridges` is then the PAIRED bridge(s) from each missed USER-SIDE word to the real ENGLISH code identifiers it maps to (e.g. `{"userWord":"efetivar","codeTerms":["effectivate","settle","confirm"]}`) — a pair, not a flat list, so the persist knows `efetivar → effectivate`, never a cartesian. Both default to `true` / empty when absent (an old reply carries no retrieval concern).
   - `requeryTerms` is the SPECULATIVE re-query set — the likely code-side wording of the missed concept (a translation/synonym, as the code would name the ACTION). Distinct from `requeryBridges`: it is a THROWAWAY search, so the judge GUESSES here even when it has no grounded bridge (a pure synonym gap where the implementer surfaces in NEITHER the name anchors nor the purpose candidates). A wrong term costs one deterministic lookup, never the lexicon — so this is the term that re-surfaces a target the name index missed AND no shown candidate implements. Empty only when `centralFound` is `true`.
3. **Act on the verdict.**
   - **`centralFound: false` → PURPOSE-SEARCH first (deterministic recall), then RE-QUERY + PERSIST (auto-heal).**
     0. **Purpose-search + judge pick — the recall the name-index structurally cannot reach.** Run `mustard-rt run purpose-search --intent "<the missed central concept, in the user's own words>" --model .claude/grain.model.json`. It returns the files whose LLM-written `purpose` summary bridges the concept — exactly the methods whose NAME diverged from the request (field-measured: PT `efetivar`→`EffectivateAsync`, `dar baixa`→`WriteOffAsync` — name-match recall 0/10). The purpose index is RECALL, not precision: a business feature's vocabulary recurs across many files (the service that PERFORMS the action, the page that DISPLAYS it, the repo that stores it), so do NOT read the top hit blindly. **Pick explicitly:** run `mustard-rt run purpose-judge-render --intent "<the missed concept>" --model .claude/grain.model.json` and dispatch it to Sonnet (same one-call judge tier) — it returns `{"picks":[<file>],"reason":"..."}`, the file(s) whose purpose PERFORMS the action (not the one that shows/stores it). Read the picks. `picks:[]` (no candidate implements it) → fall through to the re-query below. Requires a purpose-enriched scan (`enrich-purpose`); an empty search → re-query below. See [[scan-enrich-purpose]].
     1. **Re-query** with the corrected vocabulary — the union of `requeryTerms` (speculative) and the flattened `codeTerms` across `requeryBridges`, joined: `mustard-rt run feature --intent "<all terms joined>"`. `requeryTerms` is what closes a PURE synonym gap where purpose-search ALSO missed (the implementer's purpose words the action differently than the user did) — the judge's guess at the code-side wording re-surfaces it even with no grounded bridge. Use the re-query's deterministic anchors for the right flow; only THEN proceed to `dropped`/`route`/`scope` against the re-queried answer. A `strong` reason is **NOT** trustworthy when the central concept missed: `centralFound` OVERRIDES the aggregate `reason`. (Re-query, do not just drop — the right files were never in the first answer.)
     2. **Persist ONLY when DOUBLY confirmed.** A `strong` re-query proves the term EXISTS, NOT that it means the RIGHT thing — `import` matches the real `ImportOfxAsync` flow (an OFX bank-statement import — a DIFFERENT feature, NOT a programming keyword) and `user` matches the auth/login user, both `strong` in the WRONG sense, and would poison the lexicon (field-proven: `importacao→import`, `colaborador→user`). So persist only when BOTH hold: (a) the re-query `report.reason` is `strong`, AND (b) its top anchors land in the SAME domain the request is about (they overlap the concept's kept anchors / its concern), not a different flow. Then write `requeryBridges` (already in the `{userWord, codeTerms}` proposal shape) to a temp file and run `mustard-rt run lexicon-enrich --apply <file>`, surfacing a one-line note (e.g. `auto-bridged: efetivar→effectivate (confirmed)`). The next identical query then hits the deterministic `lexicon` tier — no LLM call.
     3. **Do NOT persist** when the re-query is not `strong`, OR when it lands on a DIFFERENT flow than the request (the term matched in the wrong sense). When unsure, PROPOSE rather than auto-write — surface the bridge + its re-query anchors and let the orchestrator/user confirm (the `lexicon-suggest --accept` pattern). A wrong bridge mis-directs EVERY future query, so silence beats a bad write.
   - `centralFound: true` (or absent) → proceed normally.

   **Principle:** the semantic judge is the SOURCE of new bridges; the lexicon is a CACHE the judge populates on confirmed hits — so the deterministic path covers more vocabulary over time and the LLM fires only on genuinely-new words. Two guards keep the cache clean: the judge returns an EMPTY `requeryBridges` for net-new concepts and same-word-different-sense matches (a feature to BUILD is not a word to bridge), and the persist gate writes only when the re-query lands in the request's own domain — a `strong` re-query proves a term EXISTS, never that the mapping is RIGHT. The `--apply` anti-hallucination gate only checks the target is mined; it does NOT check sense, which is why the domain check above is required.
   - **`dropped`** → remove those anchors from what you read; they are incidental (a far-layer / ambiguous-term match). Never open them.
   - **`route` = `"task"`** → the real work is single-layer and small: it does NOT need the pipeline. Run it as **`/mustard:task`** (lean — no spec, no wave, no QA gate) on the KEPT anchors, and STOP the heavier flow. *(This is the −81%-turns lever: most enhancements land here.)*
   - **`route` = `"feature"`** → continue the feature pipeline. **`scope` = `"light"`** → inline EXECUTE; **`scope` = `"full"`** → open `feature/full-plan.md` and follow PLAN.
   - **`concerns`** → each object is its OWN unit of work, scoped to its `anchors` (never the flat list): `/feature` → a unit in the three-natures split (Full: a wave); `/task` → its own dispatched action; `/bugfix` → its own diagnose+fix.
4. **Deterministic fallback (never block the flow).** If the validator is unavailable, the dispatch errors, or its reply does not parse into an object with a `route`, DROP the verdict and proceed from the flat pruned `anchors` + the DETERMINISTIC scope (fill the census → `spec-draft` → `plan-prepare`) exactly as you would without it. The validator only ever SHARPENS — its absence degrades to the deterministic path, never to a stall.

## INTENT hygiene — the rule that makes or breaks the validator
**`<INTENT>` MUST be the user's actual REQUEST — their own words / phrasing of WHAT they want — the SAME rich intent you passed to the digest (the user's content words + your code-vocabulary translation), NOT a stripped list of isolated code terms.** The validator judges meaning (is this `card` the UI card or the credit card?); given only bare terms it loses the context that disambiguates, and prunes/routes wrong. Never lapidate the request away before the validator.

## Invariant
The scan digest is **100% deterministic**; the validator is the **ONLY AI step** and lives in the **ORCHESTRATION**, never inside the scan. `digest-validate-render` does pure deterministic assembly (reuses the digest retrieval + project span, emits a byte-stable prompt, calls no model); the JUDGEMENT is the dispatched Sonnet agent's, run one layer above the scan. It SUBSUMES the older concern-split judge — its `concerns` field carries the same partition, so flows run THIS step, not a second judge.
