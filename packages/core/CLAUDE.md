# packages/core — `mustard-core`

Subproject guard for the shared library crate. Auto-loaded when working under `packages/core/`.

## What this is

The `mustard-core` crate — the shared Rust library consumed by `mustard-cli` and
`mustard-rt`. It holds cross-cutting primitives, not commands:

- `config.rs`, `env.rs` — configuration and environment-mode resolution
- `error.rs` — shared error types
- `knowledge.rs`, `metrics.rs` — knowledge-base and telemetry models
- `io/` — the SQLite event store (`sqlite_store.rs`; schema embedded from
  `sqlite_schema.sql` via `include_str!`)
- `model/` — shared data models

## Stack

- **Language:** Rust 2024, min Rust 1.85
- **Key deps:** `serde` / `serde_json`, `thiserror`, `rusqlite` (bundled)
- **Dev:** `insta` (snapshots), `tempfile` (IO tests)

## Commands

```bash
cargo build -p mustard-core      # compile

cargo clippy -p mustard-core     # lint (pedantic + unwrap_used = deny)
```

## Guards

- This is a **library** — no `main.rs`, no side effects at import. Keep it free of
  CLI/hook-specific logic; that belongs in `mustard-cli` / `mustard-rt`.
- A public API change here ripples into both consumer crates — build the whole
  workspace (`cargo build`) after touching exported items.
- Code, comments and doc-comments in EN; surgical edits only.
- Never use `unsafe` — `#![forbid(unsafe_code)]` is active.
- Every fallible function returns `Result<T>`, never panics.
- Keep `Error::NotFound` distinct from `Error::Io`; use `fail_open` for silent degradation.
- Boundary structs accepting external JSON must use `#[serde(flatten)] raw: Value`.
