01 · Conceito

O que é Mustard?

Capítulo um

Imagine que o Claude Code é um cozinheiro talentoso — mas que adora pular o mise en place: começa a fritar antes de cortar, esquece o tempero, manda pro pass sem provar. Mustard é a brigada inteira em forma de comandos.

02 · Instalação

Setup em trinta segundos

Capítulo dois

Node 18+, Claude Code, terminal aberto. Nada mais. Você roda o setup uma vez e nunca mais — toda inteligência vive na pasta .claude/.

npm install -g mustard-claude
cd meu-projeto
mustard init
ComandoPara que serve
mustard initCria .claude/, instala RTK, gera mustard.json
mustard updateAtualiza arquivos do core sem mexer nos seus customizados
mustard auto-updateMesmo que update, sem perguntas
Atenção

Rode dentro de um repositório Git já inicializado. Sem Git, a auto-detecção de subprojetos não funciona.

03 · Reconhecimento

/scan — o olhar inicial

Capítulo três

Antes de pedir qualquer coisa séria, o Claude precisa conhecer seu código. O /scan é a apresentação formal: ele lê o projeto, mapeia entidades, identifica padrões e gera fichas técnicas.

É como o primeiro dia de um novo dev na empresa. Ele lê o README, abre o banco, percorre as pastas, anota convenções. Você não pede uma feature nova logo — você deixa ele se ambientar.
/scan                # tudo (incremental por hash)
/scan api            # só subprojeto "api"
/scan --force        # ignora cache, regenera tudo

O que ele entrega

  • entity-registry.json — dicionário das suas entidades
  • commands/{stack,patterns,guards,recipes}.md — fichas que o Claude consulta
  • Skills e agents auto-gerados pro seu stack
Erro de iniciante

Pular o /scan e ir direto pro /feature. O hook enforce-registry bloqueia. Sem o scan, o Claude está cego.

Dica

Rode --force depois de mudanças grandes (refatorar pasta, novo módulo). No dia a dia, o incremental basta.

04 · Pipeline principal

/feature — o coração

Capítulo quatro

Toda mudança que toca código de produção (schema, API, UI) passa por aqui. Você fala o quê; o pipeline cuida do como. O escopo é detectado automaticamente.

Fase 1
Analyze

Lê código relevante, mapeia camadas

Fase 2
Plan

Spec + Acceptance Criteria

Fase 3
Execute

Implementa por waves

Fase 4
QA

Roda os AC um a um

Fase 5
Close

Arquiva & aprende

Detecção automática de escopo

TamanhoQuandoFluxo
Light1–2 camadas, ≤5 arquivos, padrão conhecidoAnalyze → Execute → QA → Close
ExtendedEntidade já existe, ≤8 arquivosAnalyze → Execute → QA → Close
Full3+ camadas, nova entidade ou padrãoAnalyze → Plan → /approve → Execute → QA → Close
Light é trocar uma lâmpada — você vê, troca, acabou. Full é reformar a cozinha — precisa de planta (Plan), aprovação do dono (/approve), execução supervisionada e vistoria final (QA).

Acceptance Criteria — a vistoria

## Acceptance Criteria
- [ ] AC-1: build passa — Command: `npm run build`
- [ ] AC-2: campo phone salva — Command: `node scripts/test-phone.js`
- [ ] AC-3: type-check limpo — Command: `npm run typecheck`
Windows

AC rodam no cmd.exe. Evite $(), [ ], for, test. Use node -e "…" ou bash -c "…".

05 · Diagnóstico

/bugfix — detetive com prazo

Capítulo cinco

Bug é diferente de feature: você sabe o sintoma, não a causa. O /bugfix investiga sozinho — Grep, Glob, leituras dirigidas — com orçamento limitado de movimentos.

/bugfix Login button não responde no mobile
/bugfix NullReferenceException em OrderService.Calculate
/bugfix Migration 042 quebra em staging
É um detetive com prazo — vinte movimentos para encontrar o culpado. Se acabar o tempo, ele pede ajuda. Se descobrir, corrige e testa.

Como descrever bem

BomRuim
"Carrinho zera ao trocar de aba. Erro: Cannot read property 'items' of undefined." "Adicione localStorage no carrinho." (isso é proposta, não sintoma)
Dica

Sempre que tiver, cole a mensagem de erro literal. Mustard cacheia a "raiz" e usa para retry inteligente em tentativas seguintes.

06 · Controle

Aprovar & retomar

Capítulo seis

Em features Full, você é o chef. Depois do Plan, você prova e aprova. Se a sessão cair, você retoma de onde parou.

/approve

/approve              # vai abrir nova sessão para EXECUTE
/approve --resume     # executa na MESMA sessão (use em Light)
Por quê duas sessões

Em features grandes, EXECUTE consome muito contexto. Sessão limpa = qualidade alta. Em Light, é desperdício — use --resume.

/resume

Sessão caiu? Aprovou e abriu janela nova? /resume detecta o pipeline ativo e pergunta:

  • Continuar (leve) — reaproveita o estado, segue de onde parou
  • Reanalisar (completo) — refaz scan e contexto antes de seguir

Se uma chamada de agente falhou nos últimos dez minutos, ele tenta sozinho de novo.

07 · Fechamento

QA & /complete — a vistoria

Capítulo sete

Antes do prato sair pro pass, o chef prova. Antes do pipeline fechar, os Acceptance Criteria são executados. Falhou? Não fecha.

/qa                              # spec ativa mais recente
/qa --spec linked-services       # spec específica
/complete                        # fecha tudo

Checklist obrigatório

  1. Review aprovado — alguém (ou skill code-review) deu o ok
  2. Build verde — compilou e passou type-check
  3. Mudanças batem com a spec — sem escopo expandido sem aviso
  4. Zero issues críticas — bloqueadores resolvidos
  5. Sem regressões — testes existentes ainda passam
Bloqueio comum

Tentar /complete sem QA passar. O hook close-gate trava — rode /qa antes.

08 · Atalhos

/task — o cardápio rápido

Capítulo oito

Nem tudo merece pipeline completo. Para análise, refactor cirúrgico, docs ou auditoria, use /task — pratos do dia delegados.

analyze ★ leve

Exploração rápida com Explore agent. Use para mapear área desconhecida.

audit ★★ médio

Checklist de qualidade — a11y, copy, design, i18n.

review ★★★ premium

Code quality completo — SOLID, security, performance.

refactor ★★★ premium

Plan → você aprova → implementação cirúrgica.

docs ★★ médio

Geração de documentação a partir de código.

implement ★★ médio

Mudança cirúrgica em arquivo específico, com build inline.

compare ★ leve

Cruza implementações entre subprojetos do monorepo.

Não confunda

Mudou schema + API + UI? É /feature. Renomear um campo num arquivo? /task implement.

09 · Versionamento

/git — o cinto de segurança

Capítulo nove

Opera com base no mustard.json. Recusa operações perigosas, mantém histórico linear, ignora arquivos efêmeros.

ComandoEfeito
/git syncPull rebase do branch pai na atual
/git commitCommit (sem push)
/git pushsync + commit + push
/git mergePromove branch atual ao pai (ff-only)
/git merge mainCascade completo: branch → dev → main → branch
Você não vê o cinto trabalhando — mas se freia forte, ele segura. /git recusa commit em main, ignora .agent-state/, bloqueia force-push. Os erros que ele impede são o valor.
Fluxo recomendado

Trabalhe em dev_<seu-nome>. Quando pronto, /git merge main propaga tudo de uma vez.

12 · Enforcement

Hooks — guarda-corpos invisíveis

Capítulo doze

Vinte e três hooks atuam silenciosamente. A maioria avisa, alguns bloqueiam. Airbag e ABS — você não os vê, mas estão ali.

HookO que fazComo contornar
bash-native-redirectBloqueia grep/cat/ls/find no BashUse Grep/Glob/Read nativos
tool-use-counterLimita Explore agent a 15–20 buscasReformule o prompt do agente
enforce-registryExige /scan antesRode /scan
close-gateBloqueia /complete sem QARode /qa antes
spec-size-gateAvisa quando spec fica enormeQuebre em pipelines menores
model-routing-gateModelo errado para a tarefaHaiku=Explore · Sonnet=bug · Opus=feature

Modos: off · warn · strict

Hooks heurísticos (que adivinham) ficam em warn. Só sensores reais (build, QA) bloqueiam. Filosofia do projeto: fricção sem ganho é perda.

13 · Custo

RTK — token economy

Capítulo treze

Filtro de saída pra comandos shell. Reduz outputs verbosos em até 99% mantendo o que importa. Prefixe TODO comando com rtk.

rtk git add . && rtk git commit -m "msg" && rtk git push
rtk npm run build
rtk vitest run
rtk gh pr view 123

14 · Receita completa

Workshop — do zero ao merge

Capítulo catorze

Cenário: adicionar campo phone à entidade User num projeto Next.js. Pipeline Light, cerca de dez minutos.

  1. /scan — bootstrap inicial (só na primeira vez)
  2. /status — confirma branch, build, registry
  3. /feature add-phone-to-user — Mustard detecta Light scope automaticamente
  4. Aguarde EXECUTE — sem PLAN porque é Light
  5. /qa — roda build, type-check e testes (3 AC)
  6. /complete — fecha o pipeline, arquiva spec em completed/
  7. /git push — sincroniza, commita e empurra
  8. /review — revisão automática do PR
  9. /git merge main — cascade até produção
  10. /stats — veja Pass@1 e quanto economizou em tokens
15 · O que evitar

Top 10 armadilhas

Capítulo quinze

Erros que aparecem nos primeiros dias. Memorize esta lista — é a curva de aprendizado de quem usa Mustard há semanas, condensada.

  1. Pular o /scan inicialRegistry vazio bloqueia tudo — bootstrap é obrigatório.
  2. Trabalhar direto em mainO /git recusa. Crie dev_<seu-nome> sempre.
  3. Tentar /complete sem QAO hook close-gate bloqueia. Rode /qa antes.
  4. Forçar modelo barato em featureQualidade despenca, retries explodem. Não economize aqui.
  5. Editar pipeline-states manualmenteCorrompe estado. Use /resume sempre.
  6. Commit em arquivos efêmeros.metrics/, .agent-state/ são runtime — nunca vão pro repo.
  7. Specs gigantesO spec-size-gate avisa. Quebre em features menores.
  8. POSIX em AC no WindowsUse node -e ou bash -c. Evite [ ], $(), for.
  9. Confundir knowledge com auto-memoryUm é projeto, outro é por usuário — camadas distintas.
  10. /feature para refactor puroDesperdício de pipeline. Use /task refactor.
