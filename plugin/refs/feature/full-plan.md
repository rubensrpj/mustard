## DECOMPOSE

From the insumos + anchors, split the request into three natures (this is the judgement grain cannot make):

- **Units with precedent** ‚Üí each maps to a matched slice. Only a **net-new** unit (it CREATES an entity that does not exist yet) gets a `scan spec` compile (with `--like <sibling>` when one exists) ‚Äî the compiler emits a *create* mold. An **enhancement** unit (modifies an existing entity/behavior) skips `scan spec` and consumes the feature digest's anchors instead (the `context_enrichment` that `spec-draft` pre-fills into the scaffold's `context` section).
- **Cross-cutting invariants to obey** ‚Üí contracts/hubs the repo already enforces (e.g. an injected `ICurrentTenant`); pass each via `scan spec --invariant <Name>` so the draft anchors the real wiring. NEVER invent the mechanism ‚Äî mirror the anchored consumers.
- **Net-new gaps** (no precedent; `miss` after a repo-vocabulary re-query) ‚Üí surface as a design decision; do not let a `scan spec` draft's "deterministic" framing imply a unit that has no precedent is safe to clone.

**Concerns come from the digest ‚Äî deterministic, no judge.** When the scan's deterministic `concerns` show ‚â•2, each concern is one unit in the three-natures split above, and maps to its own wave. Use that partition directly; a single concern needs no partition at all.

## PLAN

‚Üí `./spec-language.md` (header translation, narrative rules, Component Contract). ‚Üí `./wave-decomposition.md` (the `--plan` schema).

Resolve Lang via cascade (`meta.json#lang` ‚Üí `mustard.json#specLang` ‚Üí AskUserQuestion once) ‚Äî hold the resolved value for `spec-draft --lang` (step 2 persists it to `meta.json`).

PLAN materialises the spec in a **fixed order** ‚Äî every artifact exists before the next step consumes it:

1. **Lapidate the body (no file yet).** Per **net-new** unit (creation): `mustard-rt run scan spec --entity {Unit} [--like {Sibling}] [--invariant {Contract}] [--ops create,...]` ‚Üí a draft carrying the auto-chosen pattern menu, per-project sections, the anchors, and acceptance criteria. An **enhancement** unit skips `scan spec` (the compiler only emits a *create* mold) ‚Äî lapidate it from the feature digest's anchors (the `context_enrichment` `spec-draft` folds into the scaffold's `context` section). Read the draft + its anchors + the client request; resolve the bifurcation, prune, add domain rules, mark assumptions ‚Äî **in the project's language/tone** (`mustard.json#specLang`/`tone`). The draft's project-unit sections ARE the wave/agent decomposition. Hold the lapidated body in context; do **not** write a file here.
2. **Materialise the scaffold.** `mustard-rt run spec-draft --intent "<request>" --scope full --lang <bcp47> [--waves N] [--query-terms "<repo-vocabulary terms>"]` writes `.claude/spec/{slug}/spec.md` + `meta.json` (slug born from `--intent`; `meta.json` is the single lifecycle source; the `context` section is pre-filled with the scan anchors/slices). **When the user's raw words alone came back `weak`/`none` in ANALYZE** (so it was your code-vocabulary translation that carried the strong match), pass `--query-terms` with the comma-separated code-terms that produced the strong report ‚Äî without it the draft re-tokenises the raw intent, repeats the weak query, and the enrichment withholds itself (a weak answer never materialises anchors into the scaffold). This is the **only** writer of the spec scaffold ‚Äî never hand-write `spec.md` with the Write tool. **Routing gate (the machine enforces the economy):** `spec-draft` re-classifies the spec.md it wrote and AUTO-REBAIXA a `--scope full` the deterministic signals do not justify (single-layer, ‚â§5 files, no net-new) down to light ‚Äî it rewrites `meta.json#scope` (the source of truth) and reports `scopeDowngraded:{from,to}`; trust that over the `--scope` you passed (and if it downgrades, return to the thin SKILL's Light EXECUTE path). `--force-scope` honours the requested full anyway (recorded as an auditable `pipeline.scope.override` event). The gate is a no-op when the census is still a placeholder (`filesSectionEmpty`) ‚Äî fill `## Files` BEFORE the draft for it to act.
   Right after the scaffold exists, run `mustard-rt run digest-adherence-finalize --spec {slug}` (the slug just born from `spec-draft`). Fire-and-forget telemetry: it folds the session's events into one `analyze.digest.summary` attributed to the new spec; it never blocks and its output requires no action ‚Äî continue to step 3 immediately.
3. **Fold the body in ‚Äî into the plan JSON, not by hand after the scaffold.** `Edit` the lapidated bodies from step 1 into the scaffold's parent Plan-layer sections (`entities`, `files`, optional `component-contract` UI-only, `tasks`, `dependencies`, `boundaries`). Use the `decompose`/`waves` that **`plan-prepare`** already returned in ANALYZE (multi-wave on its signals; otherwise one wave) ‚Äî no separate `scope-decompose` call. The lapidated `scan spec` project-unit decomposition becomes the **per-wave body of the plan JSON** ‚Äî each wave carries `tasks` (checklist), `files` (census), and `acceptance` (AC) arrays. **Before materialising**, validate/derive the plan's `depends_on` with `mustard-rt run wave-dependency --plan <plan.json>` (file-based input ‚Äî it accepts the plan JSON's `{waves:[...]}` shape, unioning per-wave `files`; never pipe stdin, it does not survive `rtk`).
