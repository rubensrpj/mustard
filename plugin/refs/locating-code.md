# Locating code — literal → grep, concept → digest, rules → Guards

Loaded by every flow that finds code (`/feature`, `/task`, `/bugfix`, ad-hoc location). **Single source of truth — reference this file, never copy the prose.** Pick the tool by WHAT you already know:

- **Orienting (which subproject?)** → the terrain is ALREADY in your window: the orientation census (`mustard-rt run orient`) is injected at session start (each subproject `name · kind · files · layer`). Never grep just to orient. There is no per-prompt file suggestion (that guesser was removed after 1 useful hit in 17) — pick a tool below by what you know.
- **A LITERAL token (exact symbol, error string, file glob)** → `grep`/`glob`. Exact, instant, complete — never spend a digest call on a token you already have.
- **A CONCEPT whose name may diverge** ("settle a payable" → `WriteOffAsync`, "efetivar" → `EffectivateAsync`) → the deterministic digest: `mustard-rt run feature --intent "<lapidated code-shaped terms + the user's content words>"` — a BM25 ranking over the mined `grain.model.json`, no model call.
  - **Lapidate the request into code-shaped terms yourself:** strip the glue (prepositions/articles — content words only), translate into the code's vocabulary, shape it how code NAMES things — verbs infinitive (`create`/`list`/`update`), collection nouns plural (`clients`/`contracts`/`receivables`); when unsure include both forms (the digest dedups). Code-shaped terms hit the **EXACT** tier, not `stem` (where the noise lives).
  - It returns ~12 anchors with per-anchor provenance. **Prune by provenance** (`anchorsDetail` shows each anchor's matched terms — drop the tangential, keep the central), then **READ only the survivors — the digest finds WHERE to look, it does not replace reading.** On a `weak`/`none` report it returns the `candidates` term index (the repo's REAL vocabulary): sharpen against it and re-call — a `miss` is NOT "absent". Persist a confirmed bridge via `mustard-rt run equivalence-learn --term <missed> --tokens <code-terms>`.
- **The LAYER's rules (conventions, do/don'ts)** → the subproject's `## Guards` in its `CLAUDE.md` — auto-loaded the moment you work under that subproject; no pipeline needed.

**Shape the query — lead with the WORK, not the DATA.** When the task is bound to ONE layer (a frontend datatable cell, a migration, an endpoint), lead the `--intent` with the terms that name the WORK/layer (`datatable`, `column`, `cell`, `wrap`, `tooltip`, `render`) over the shared DOMAIN nouns the data carries (`cpf`, `cnpj`, `document`, `validation`). A domain noun matches in EVERY layer, and the plumbing layer usually has MORE occurrences of it — so a data-noun-led query lets the wrong layer out-score your target on raw frequency. The work/structure vocabulary DISCRIMINATES the layer; the domain noun is the noise that drags in the wrong-layer plumbing.

Digest recall is strong but NOT perfect — ALWAYS verify by READING the candidates; never act on the top hit blind. The `grain.model.json` index is built by `/scan`; if absent, run `/scan` first — grep alone cannot bridge a vocabulary gap.
